# MVTI 앱 제품·과제 통합 설계안

## 결론부터 정리

이번 과제에서는 **영화 데이터 자체를 CRUD하는 앱**보다, 사용자의 **영화 반응 기록**을 CRUD하는 앱으로 정의하는 것이 가장 안전합니다.

> 사용자는 영화 풀에서 영화를 골라 `좋아요·싫어요·보고 싶어요` 기록을 추가하고, 목록에서 조회하고, 반응이나 메모를 수정하고, 기록을 삭제한다. 이 데이터로 MVTI 유형과 추천 결과가 계산된다.

이 구조라면 과제의 핵심인 목록 조회·추가·수정·삭제가 사용자 경험에 자연스럽게 포함됩니다. 과제 안내가 요구하는 데이터 구조 정의, 컴포넌트 분리, 입력·빈 상태·오류 처리, 테스트, AI 활용 기록, 오류 해결 기록, README, 캡처와 보안 점검까지 모두 명시적으로 증명할 수 있습니다. 

동시에 실제 서비스로 발전할 수 있도록 다음 2개 층으로 설계합니다.

* **과제 핵심층:** React + Vite + LocalStorage 기반 반응 기록 CRUD
* **서비스 확장층:** Supabase Auth·DB·RLS, 일일 추천, 유사 사용자 추천

인증이나 외부 API 오류가 발생해도 필수 CRUD가 깨지지 않도록 **LocalStorage 버전부터 완성한 뒤 Supabase를 연결**해야 합니다.

# 1. MVTI 유형 정의

## 1-1. MVTI의 정확한 정의

**MVTI, Movie Viewing Taste Indicator**는 성격을 진단하는 심리검사가 아니라, 사용자가 선택한 선호·비선호 영화에서 반복적으로 나타나는 **관람 취향 패턴을 4개 축으로 요약한 영화 취향 유형**입니다.

MBTI는 사용자가 직접 입력하는 참고 정보이고, MVTI는 실제 영화 선택으로 계산합니다.

따라서 결과 문구도 다음처럼 구분해야 합니다.

* 잘못된 표현: “당신은 감성적인 성격입니다.”
* 올바른 표현: “당신은 영화에서 감정과 관계가 중심이 되는 이야기를 상대적으로 더 선호합니다.”

기존 조사 자료의 MBTI별 영화 목록은 초기 가설이나 콜드 스타트 규칙으로는 쓸 수 있지만, 유형당 영화 수가 적고 MBTI와 영화 취향을 고정적으로 연결한다는 한계가 있습니다. 조사 자료 자체도 장기적으로는 외부 성격 DB보다 자체 사용자 반응과 평점을 축적하는 방향을 권고합니다. 

---

## 1-2. MVTI의 4개 축

각 축은 0~100점의 연속 점수로 저장하고, 결과 화면에서 우세한 쪽의 문자를 선택합니다.

| 축       | 왼쪽 성향             | 오른쪽 성향          | 판별하는 취향                            |
| ------- | ----------------- | --------------- | ---------------------------------- |
| **F–R** | Fiction, 비현실·세계관형 | Reality, 현실·생활형 | 판타지적 세계와 현실 기반 이야기 중 어디에 더 몰입하는가   |
| **H–L** | High Energy, 고자극형 | Low Tempo, 저자극형 | 빠른 전개·강한 자극과 느린 호흡·관찰 중 무엇을 선호하는가  |
| **E–I** | Emotion, 감정형      | Idea, 사유형       | 관계·감정적 카타르시스와 개념·구조·논리 중 무엇을 중시하는가 |
| **O–C** | Open, 여운형         | Closure, 완결형    | 모호함·열린 결말과 명확한 해결·완결 중 무엇을 선호하는가   |

예를 들어 `M-FHIC`는 다음 취향입니다.

* 비현실적 세계관
* 빠르고 강한 전개
* 감정보다 아이디어와 구조
* 명확한 결말과 해결

`M-` 접두사를 붙여 MBTI와 혼동되지 않게 합니다.

## 1-4. MVTI 계산 방식

각 영화는 다음과 같은 취향 벡터를 가집니다.

```text
영화 A = {
  fictionReality: 80,
  energyTempo: 65,
  emotionIdea: 30,
  openClosure: 75
}
```

각 점수는 0~100이며, 예를 들어 `fictionReality: 80`은 Fiction 쪽에 가깝다는 뜻입니다.

사용자 반응 가중치는 다음처럼 처리합니다.

| 반응     |       계산 |
| ------ | -------: |
| 좋아요    |       +1 |
| 정말 좋아요 |     +1.5 |
| 싫어요    |       -1 |
| 정말 싫어요 |     -1.5 |
| 보고 싶어요 | 취향 계산 제외 |
| 지금은 아님 | 취향 계산 제외 |
| 본 적 없음 |       제외 |

중요한 점은 **“지금 보고 싶지 않음”을 “싫어함”으로 계산하지 않는 것**입니다. 컨디션과 영구 취향을 분리해야 추천이 망가지지 않습니다.

초기 계산 원칙은 다음과 같습니다.

```text
최종 취향 벡터
= 영화 선택 근거 85%
+ MBTI 기반 초기 참고값 최대 15%
```

MBTI 참고값은 선택 기록이 늘어날수록 감소시킵니다.

* 6개 선택: MBTI 참고값 최대 15%
* 12개 선택: 5% 이하
* 20개 이상: MBTI 영향 제거

따라서 사용자의 실제 행동이 MBTI 고정관념보다 항상 우선합니다.

## 1-5. 결과의 신뢰도

유형 코드만 보여주면 지나치게 단정적으로 보일 수 있으므로 반드시 신뢰도를 함께 표시합니다.

```text
현재 MVTI: M-RLIO
분석 신뢰도: 68%
분석 근거: 선호 7편, 비선호 4편, 8개 장르
```

신뢰도는 다음 세 요소로 계산합니다.

* 선택한 영화 수
* 선택 장르의 다양성
* 각 축이 50점에서 얼마나 멀리 떨어져 있는가

6편으로도 임시 결과를 제공하되, 12편 이상에서 “안정적 유형”으로 표시하는 것이 적절합니다.

# 2. 영화 풀 확보 계획

## 2-1. 권장 초기 규모

단순히 장르별 10편을 채우는 수준보다 다음 규모가 적절합니다.

> **초기 고정 풀: 고유 영화 240편**

운영 장르 18개에 각각 **주 장르 기준 최소 12편**을 배정합니다.

| 운영 장르                         |
| ----------------------------- |
| 액션, 모험, 애니메이션, 코미디, 범죄, 다큐멘터리 |
| 드라마, 가족, 판타지, 역사, 공포, 음악·뮤지컬  |
| 미스터리, 로맨스, SF, 스릴러, 전쟁, 서부극   |

영화는 여러 장르를 가질 수 있지만 장르별 최소 수량을 셀 때는 `primaryGenre` 하나만 사용합니다. 그렇지 않으면 한 편의 액션·SF·모험 영화가 세 장르 수량에 중복 집계되어 풀의 실제 다양성이 떨어집니다.

* 18개 장르 × 주 장르 12편 = 216편
* 장르 경계형·실험형·추천 다양성용 영화 = 24편
* 총 240편

첫 화면에서 움직이는 포스터에는 이 중 **인지도가 높고 축 구분력이 좋은 48~64편**만 사용합니다.

---

## 2-2. 풀의 균형 기준

240편 전체에 다음 제약을 적용합니다.

### 지역 균형

* 한국 영화: 약 25%
* 영어권 영화: 약 45%
* 기타 비영어권 영화: 약 30%

단, 서부극처럼 한국 영화 확보가 어려운 장르에 억지로 동일 비율을 적용하지 않고 전체 풀에서 맞춥니다.

### 시대 균형

* 1990년 이전 및 고전: 10~15%
* 1990년대: 10%
* 2000년대: 20%
* 2010년대: 30%
* 2020년대: 25~30%

### 인지도 균형

* 대중적 히트작: 40%
* 중간 인지도 작품: 40%
* 숨은 명작·취향 탐색용 작품: 20%

### 중복 방지

* 동일 프랜차이즈는 기본적으로 최대 2편
* 동일 감독 편중 제한
* 리메이크와 원작은 별도 목적이 있을 때만 함께 포함
* 포스터가 없거나 한국어 제목 식별이 어려운 영화 제외
* 지나치게 낮은 평가 수의 영화는 초기 온보딩 풀에서 제외

---

## 2-3. 영화별 저장 필드

```ts
Movie {
  id: string
  tmdbId: number
  titleKo: string
  originalTitle: string
  releaseYear: number
  primaryGenre: string
  genres: string[]
  posterPath: string
  overviewKo: string
  originalLanguage: string
  productionCountries: string[]
  runtime: number | null

  fictionReality: number
  highLowTempo: number
  emotionIdea: number
  openClosure: number

  moodTags: string[]
  themeTags: string[]
  popularityTier: "mainstream" | "mid" | "niche"
  onboardingEligible: boolean
  source: "tmdb"
  poolVersion: string
}
```

영화의 4개 축 점수는 장르만으로 정하면 안 됩니다. 같은 SF라도 감정 중심 영화와 개념 중심 영화가 다르기 때문입니다.

축 점수는 다음 자료를 함께 봅니다.

* 장르
* 줄거리
* 키워드
* 전개 속도
* 주제
* 결말의 개방성
* 감정·관계 비중
* 세계관의 비현실성

초기 240편은 AI로 1차 라벨링하되 사람이 검토합니다. 최소한 축 점수가 20 이하 또는 80 이상처럼 극단적인 영화와 온보딩용 영화는 수동 검토해야 합니다.

## 2-4. 데이터 구축 절차

### 1단계: 후보 600편 수집

장르·국가·연도·인지도 조건별로 후보를 넓게 수집합니다.

### 2단계: 자동 필터링

* 중복 제거
* 포스터 없는 항목 제거
* 지나치게 낮은 평가 표본 제거
* 성인 콘텐츠 제외
* 프랜차이즈 과다 제거
* 한국어 메타데이터 없는 항목 표시

### 3단계: 360편 1차 선별

장르별 20편 수준의 후보 목록을 만듭니다.

### 4단계: MVTI 축 라벨링

각 영화에 4개 축 점수와 감정·주제 태그를 부여합니다.

### 5단계: 240편 최종 확정

장르 수량뿐 아니라 4개 축의 분포가 한쪽으로 몰리지 않았는지 확인합니다.

### 6단계: 온보딩용 64편 선정

많은 사용자가 알아볼 가능성이 높고, 서로 다른 축을 잘 구분하는 영화만 선정합니다.

### 7단계: 버전 관리

```text
poolVersion: 2026.07-v1
```

영화 추가·삭제 이유를 `data/CHANGELOG.md`에 기록합니다.

## 2-5. 외부 데이터 사용 원칙

과제용 비상업 프로토타입에서는 TMDB를 메타데이터와 포스터 경로의 주 공급원으로 사용하는 것이 가장 단순합니다. TMDB는 비상업적 이용 시 출처 표기와 지정 문구를 요구하며, 상업 프로젝트는 별도 상업 라이선스를 문의해야 합니다. 앱의 About 또는 Credits 영역에 TMDB 고지 문구와 로고 규칙을 반영해야 합니다. ([The Movie Database (TMDB)][1])

실시간 API를 React 브라우저에서 직접 호출하기보다, 과제 제출본에서는 미리 생성한 `movies.json`을 읽도록 설계하는 것이 안전합니다. 이렇게 하면 외부 API 장애와 키 노출이 CRUD 평가에 영향을 주지 않습니다.

MovieLens는 추천 실험과 알고리즘 검증에는 유용하지만, 해당 데이터의 README는 상업적·수익 목적 이용에 사전 허가가 필요하다고 명시합니다. 따라서 과제 연구에는 활용할 수 있으나 실제 유료 서비스의 운영 데이터로 그대로 포함해서는 안 됩니다. 실제 서비스는 자체 사용자 반응을 축적해야 합니다. ([GroupLens Files][2])

OTT 제공처 정보는 향후 TMDB의 Watch Providers 응답을 활용할 수 있지만, JustWatch 출처 표기가 필요하고 직접 재생 링크 전체를 주는 API는 아닙니다. 한국 서비스에서는 국가를 KR로 구분해 “어디서 볼 수 있는지”를 보조 정보로 사용합니다. ([The Movie Database (TMDB)][3])

# 3. React CRUD 과제 완전 충족 구조

## 3-1. 핵심 CRUD 엔터티

영화 풀은 읽기 전용 데이터로 두고, 과제의 핵심 엔터티를 다음으로 정의합니다.

> `UserMovieReaction`: 사용자가 특정 영화에 남긴 취향·관람 기록

```ts
UserMovieReaction {
  id: string
  movieId: string
  sentiment: "like" | "dislike" | "neutral"
  strength: 1 | 2 | 3
  watchStatus: "seen" | "wantToWatch" | "notNow"
  note: string
  createdAt: string
  updatedAt: string
}
```

## 3-2. CRUD 대응

| 과제 기능   | MVTI 앱 동작                   | 화면 증거            |
| ------- | --------------------------- | ---------------- |
| Read    | 내가 선택한 영화 반응 목록 조회          | 내 취향 기록 페이지      |
| Create  | 영화에 좋아요·싫어요·보고 싶어요 추가       | 포스터 선택 후 반응 추가   |
| Update  | 반응, 강도, 관람 상태, 메모 수정        | 수정 모달과 수정 후 목록   |
| Delete  | 취향 프로필에서 영화 기록 제거           | 삭제 확인 후 목록에서 사라짐 |
| Persist | LocalStorage 또는 Supabase 저장 | 새로고침 후 기록 유지     |

좋아요 버튼을 다시 누르는 단순 토글만으로 Update를 증명하려 하지 말고, 별도의 **수정 버튼과 수정 폼**을 제공해야 평가자가 CRUD를 즉시 확인할 수 있습니다.

## 3-3. 저장 구조

### 과제 필수 버전

```text
localStorage key: mvti_user_reactions_v1
```

* 최초 로드 시 JSON 파싱
* 오류 발생 시 빈 배열과 오류 안내
* 변경 시 자동 저장
* 새로고침 후 유지
* 초기화 버튼은 별도 확인 절차 적용

### 서비스 확장 버전

Supabase를 연결하면 다음 테이블이 필요합니다.

```text
profiles
movies
user_movie_reactions
mvti_snapshots
daily_recommendations
```

`user_movie_reactions.user_id`는 로그인 사용자의 ID와 연결하고, 사용자가 자신의 행만 조회·추가·수정·삭제하도록 RLS를 적용합니다. Supabase 공식 문서도 공개 스키마 테이블에는 RLS를 활성화하고 Auth와 결합해 사용자별 접근을 제한하도록 안내합니다. service role 키는 브라우저에서 사용하면 안 됩니다. ([Supabase][4])

# 4. 회원 가입과 일일 재방문 설계

## 4-1. 가입 전에 먼저 가치를 보여준다

처음부터 회원 가입을 요구하면 이탈 가능성이 높습니다.

권장 순서는 다음과 같습니다.

1. 게스트 상태로 선호·비선호 영화 최소 6편 선택
2. 임시 MVTI 코드와 추천 1편 공개
3. “프로필 저장하고 전체 분석 보기”를 통해 가입 유도
4. 가입 후 전체 결과·매일 3편 추천·기록 동기화 제공

회원 가입은 기능 접근의 장벽이 아니라, 이미 만든 취향 프로필을 **잃지 않기 위한 저장 행동**이 되어야 합니다.

---

## 4-2. 매일 접속할 이유: 오늘의 3편

매일 추천을 단순히 점수순으로 3편 보여주면 금방 단조로워집니다.

매일 다음 세 슬롯을 제공합니다.

| 슬롯          | 역할                     |
| ----------- | ---------------------- |
| **정확 추천**   | 현재 취향과 가장 가까운 안전한 선택   |
| **취향 확장**   | 한 축만 다른 인접 취향 영화       |
| **오늘의 의외작** | 덜 본 장르 중 품질 기준을 통과한 영화 |

각 추천에는 반드시 이유를 표시합니다.

```text
추천 이유
- 느린 전개와 현실적인 인간관계를 선호함
- 열린 결말 영화에 긍정 반응이 많음
- 최근 드라마 장르를 4편 연속 선택함
```

---

## 4-3. 상황형 필터

영화 취향과 오늘의 상황은 다를 수 있습니다.

추천 전 다음 중 하나를 선택하게 하면 실제 사용 가치가 커집니다.

* 오늘은 가볍게
* 깊게 몰입하고 싶어요
* 100분 이하
* 혼자 볼 영화
* 가족과 볼 영화
* 이미 본 영화 제외
* 현재 볼 수 있는 OTT 우선

이 정보는 영구 MVTI를 바꾸지 않고 당일 추천에만 반영합니다.

---

## 4-4. 추천 후 피드백

추천 카드에는 다음 반응을 둡니다.

* 저장
* 이미 봤고 좋았어요
* 이미 봤지만 별로였어요
* 지금은 아니에요
* 관심 없어요

`지금은 아니에요`는 취향 점수를 낮추지 않고, `관심 없어요`와 `별로였어요`만 부정 신호로 사용합니다.

---

## 4-5. 재방문 콘텐츠

### 매일

* 오늘의 3편
* 전날 추천에 대한 한 번의 피드백
* 새로 추가된 영화 중 취향 일치 작품

### 매주

* “이번 주 MVTI 변화”
* 선호 장르 비중 변화
* MBTI와 MVTI 차이 업데이트
* 아직 평가하지 않은 취향 경계 영화 3편

### 사용자 데이터가 충분해진 뒤

* 동일 MVTI 사용자들이 높게 평가한 영화
* 나와 반대 MVTI가 좋아하는 의외의 영화
* 유형별 극과 극 한 줄 리뷰

단, 초기에는 실제 유사 사용자 데이터가 없으므로 이를 가장해서 표시하면 안 됩니다. 초기 추천은 콘텐츠 기반으로 제공하고, 각 유형에 일정 수준의 반응 데이터가 축적된 뒤 협업 필터링을 활성화합니다.

운영 시작 기준 예시는 다음과 같습니다.

```text
해당 MVTI 사용자 30명 이상
각 사용자 유효 반응 10개 이상
각 추천 영화의 유효 반응 10개 이상
```

이 수치는 출시 후 데이터에 따라 조정할 운영 가설입니다.
