연구 대상과 단순 구현 대상을 분리해야 합니다. 현재 목록에는 **코드를 보면 바로 고칠 수 있는 항목**, **제품 구조를 다시 정의해야 하는 항목**, **별도 데이터·시장 조사가 필요한 항목**이 섞여 있습니다.

git add -A

git push origin main
git push origin v1.0

# 2. 항목 1: 포스터 누락 문제

이것은 현재 상태만으로 정확한 영화를 특정할 수 없습니다. 다만 증상상 다음 가능성이 큽니다.

### Favela Funk

* 영화가 아니라 음악·다큐멘터리·잘못된 미디어 항목일 가능성
* `poster_path`가 `null`
* TMDB ID가 잘못 매칭됨
* 이미지 URL 조합 오류

### `Sur un...`

* 제목이 UI 너비 때문에 잘린 것인지
* 실제 데이터의 제목이 불완전한 것인지
* 영화 대신 단편·TV 항목이 들어온 것인지 확인 필요

### 엑시트와 소울 사이의 아랍 문자 영화

아랍 문자가 나온다는 이유만으로 오류라고 단정할 수는 없습니다.

가능성은 다음 둘입니다.

1. 실제 아랍어 영화가 의도적으로 포함됨
2. `titleKo`가 없을 때 `original_title`을 그대로 표시한 결과

현재 데이터에서 다음 필드를 찾아야 합니다.

```js
{
  id,
  tmdbId,
  title,
  titleKo,
  originalTitle,
  originalLanguage,
  posterPath
}
```

Antigravity에게는 아직 “고쳐라”라고 지시하기보다 먼저 진단 보고를 요청해야 합니다.

```text
현재 영화 데이터 전체를 검사해 다음 항목을 표로 출력하라.

1. posterPath가 null, 빈 문자열 또는 잘못된 형식인 영화
2. 포스터 URL 요청이 실패하는 영화
3. 한국어 제목이 없어서 originalTitle이 표시되는 영화
4. 영화가 아닌 TV, 음악, 인물 또는 기타 미디어가 섞여 있는지
5. Favela Funk, "Sur un"으로 시작하는 작품, 엑시트와 소울 사이에 표시되는
   아랍 문자 작품의 정확한 데이터 레코드와 TMDB ID
6. 아직 코드를 수정하지 말고 원인과 수정 후보만 보고하라.
```

# 3. 항목 2: 홈으로 돌아가는 기능

좋은 요구입니다. 다만 “홈으로 가는 버튼”만 추가하면 부족합니다. 필요한 것은 **재탐색 루프**입니다.

권장 사용자 흐름은 다음과 같습니다.

```text
결과·추천·내 취향 페이지
→ 영화 더 고르기
→ 흐르는 포스터 홈
→ 기존 선택 표시
→ 추가 선택 또는 기존 반응 변경
→ MVTI 재계산
```

버튼 명칭은 `홈`보다 목적이 드러나는 표현이 낫습니다.

* **영화 더 고르기**
* **취향 더 알려주기**
* **포스터 다시 보기**

사용자가 표현한 “멍하니 포스터를 쳐다보는” 경험은 단순 홈 이동이 아니라 서비스의 중요한 소비 모드가 될 수 있습니다. 내부적으로는 다음 두 모드로 나눌 수 있습니다.

* **선택 모드:** 포스터를 클릭하며 취향 입력
* **감상 모드:** UI를 최소화하고 포스터 흐름을 구경

기존 선택 영화에는 다음 표시가 필요합니다.

| 기존 반응  | 포스터 표시              |
| ------ | ------------------- |
| 좋아요    | 작은 하트 또는 선택 테두리     |
| 별로     | Shrug 아이콘 또는 점선 테두리 |
| 보고 싶어요 | 북마크                 |
| 미선택    | 표시 없음               |

선택 상태는 페이지 내부 상태가 아니라 앱 전역 또는 저장소 데이터에서 읽어야 합니다. 그래야 하위 페이지에서 돌아와도 유지됩니다.

# 4. 항목 3: 영화 풀 확대

이것은 별도 연구 가치가 매우 높습니다.

단순히 영화를 많이 추가하면 다음 문제가 생깁니다.

* 사용자가 모르는 영화만 계속 노출됨
* 한 장르나 국가에 편중됨
* 포스터가 없는 영화 증가
* 같은 프랜차이즈가 반복됨
* MVTI 분류에 도움이 되지 않는 유사 영화만 늘어남
* 추천 품질보다 데이터량만 증가함

따라서 영화 풀은 최소 네 가지 층으로 나눠야 합니다.

| 층       | 역할                    | 권장 비율 |
| ------- | --------------------- | ----: |
| 대중적 기준작 | 대부분이 알아보고 취향을 표현하기 쉬움 |   35% |
| 장르 대표작  | 특정 장르 선호를 구분          |   30% |
| 한국 영화   | 국내 사용자에게 높은 식별력       |   20% |
| 마이너·발견형 | 취향 확장과 차별화            |   15% |

요청한 영화는 우선 추가 후보로 적절합니다.

* 미키 17
* 곡성
* 보헤미안 랩소디
* 월터의 상상은 현실이 된다
* 위플래쉬

다만 각 영화에는 제목·장르만이 아니라 **취향 변수 점수**가 필요합니다. 그래서 영화 풀 확대는 7A의 변수 재설계 이후에 하는 편이 맞습니다.

MovieLens의 Tag Genome은 영화마다 `atmospheric`, `thought-provoking`, `realistic` 같은 속성이 어느 정도 적용되는지를 연속 점수로 표현합니다. 2021 데이터는 9,734편의 영화에 1,084개 태그와 약 1,050만 개의 영화–태그 관련도 점수를 제공합니다. MVTI도 이런 방식으로 “영화 하나당 여러 취향 속성”을 저장하는 편이 4분법보다 확장성이 높습니다. ([GroupLens][1])

# 5. 항목 4: More 전환 애니메이션

아이디어는 좋지만, “물결처럼 촤라라락”만으로 Antigravity에게 맡기면 과장된 AI 데모형 애니메이션이 나올 가능성이 높습니다.

먼저 동작을 구체적으로 정의해야 합니다.

## 권장 동작

1. More 버튼 클릭
2. 현재 포스터 행마다 30~60ms 시차 발생
3. 앞쪽 포스터는 확대되며 옆으로 빠짐
4. 뒤쪽 포스터는 축소·회전하며 사라짐
5. 새 포스터가 반대 방향에서 물결처럼 진입
6. 전체 시간은 약 700~1,000ms
7. 애니메이션 중 연속 클릭 방지
8. 기존 선택 영화는 새 묶음에서도 표시 유지

중요한 것은 애니메이션보다 **새 영화 선택 알고리즘**입니다.

`More`는 무작위 새로고침이 아니라 다음 조건을 만족해야 합니다.

* 현재 화면 영화 제외
* 이미 여러 번 본 영화 우선순위 하락
* 선택하지 않은 장르 보강
* 대중작과 마이너작 혼합
* 이미 반응한 영화는 최소화
* 최근 노출 이력 저장

즉, 이 기능은 디자인과 추천 로직을 함께 연구해야 합니다.

# 6. 항목 5: 하위 페이지 리팩터링

“바이브코딩스럽다”는 평가는 보통 다음 증상을 뜻합니다.

* 모든 요소가 카드 안에 들어 있음
* 지나치게 많은 그라데이션
* 동일한 둥근 모서리 반복
* 아이콘과 배지가 과다함
* 설명 문구가 너무 많음
* 화면마다 다른 디자인 언어
* 데이터보다 장식이 먼저 보임
* 섹션마다 비슷한 크기의 카드가 나열됨

Claude로 모델만 바꾼다고 해결되지는 않습니다. 먼저 디자인 원칙을 문서화해야 합니다.

MVTI에는 다음 방향이 적합합니다.

> **영화 포스터와 타이포그래피가 주인공이고, UI 컨테이너는 최대한 사라지는 편집 디자인**

### 리팩터링 원칙

* 카드 수 절반 이하로 축소
* 배경 그라데이션 최소화
* 테두리보다 간격과 타이포그래피로 구획
* 결과 코드는 크게, 설명은 짧게
* 포스터 비율을 유지
* 같은 페이지에서 강조색은 한 종류
* CTA는 페이지당 하나를 기본으로
* 추천 근거는 필요할 때 펼쳐 보기

이 항목은 현재 하위 페이지 캡처를 보고 페이지별로 연구해야 합니다.

# 7. 항목 6: 선택 옵션 단순화

`싫어해요`를 `나는 별로`로 바꾸는 것은 방향이 좋습니다.

`싫어해요`는 감정 강도가 지나치게 높고, 사용자가 누르기를 망설일 수 있습니다. 다음 3단계가 가장 직관적입니다.

* **내 취향**
* **나는 별로**
* **안 봤어요**

아이콘은 다음처럼 대응할 수 있습니다.

| 옵션    | 아이콘          |
| ----- | ------------ |
| 내 취향  | Heart        |
| 나는 별로 | Shrug 또는 Meh |
| 안 봤어요 | EyeOff       |

다만 Shrug 아이콘은 사용하는 아이콘 라이브러리에 실제로 존재하는지 먼저 확인해야 합니다. 존재하지 않으면 `Meh`, `ThumbsDown`, 사용자 정의 SVG 중 하나를 결정해야 합니다.

`보고 싶어요`는 취향 판별 응답과 섞지 말고 별도의 북마크 동작으로 분리하는 편이 낫습니다.

# 8. 최우선 연구: MVTI 변수 재설계

이 부분은 사용자의 문제 제기가 정확합니다.

기존 네 축이 다음과 같다면:

* Fiction ↔ Reality
* High energy ↔ Low tempo
* Emotion ↔ Idea
* Open ↔ Closure

4개 이진 축으로 16유형을 만들 수는 있지만, 영화 취향을 충분히 설명하기 어렵습니다. 특히 하나의 영화가 판타지이면서도 현실적 사회 문제를 다룰 수 있고, 느리면서도 감정 강도가 높을 수 있기 때문입니다.

## Rotten Tomatoes 간극은 무엇인가

Rotten Tomatoes의 Tomatometer는 승인된 전문 평론가 리뷰 중 긍정 평가의 비율이고, Popcornmeter는 일반 사용자가 제출한 평가를 기반으로 합니다. 따라서 둘의 차이는 동일한 기준으로 측정한 두 점수의 단순한 품질 차이가 아니라, **평론가 집단과 관객 집단의 긍정 비율 차이**입니다. ([Rotten Tomatoes][2])

예를 들어:

```text
Tomatometer 90
Popcornmeter 55
gap = -35
```

이 값은 “관객보다 평론가가 더 좋아한 영화”라는 사회적 수용 특성을 나타냅니다.

다만 이것을 MVTI의 핵심 축으로 직접 쓰는 것은 부적절합니다.

### 이유

* 모든 영화에 두 점수가 존재하지 않음
* 평가 참여 집단과 표본 규모가 서로 다름
* 영화 공개 시기와 인지도에 영향을 받음
* 취향이 아니라 영화의 외부 평가 결과임
* 평론가–관객 간극이 큰 이유가 영화마다 다름

따라서 RT 간극은 **영화 속성 축**이 아니라 다음과 같은 사용자 성향을 추론하는 보조 변수로 쓰는 것이 낫습니다.

> `criticalAffinity`: 평론가 선호 영화와 관객 선호 영화 중 어느 쪽에 더 반응하는가

## MVTI v2 후보 변수

4개의 이진 축 대신 연속형 취향 벡터를 권장합니다.

### 핵심 8개

| 변수                    | 낮은 쪽     | 높은 쪽         |
| --------------------- | -------- | ------------ |
| `realism`             | 환상·초현실   | 현실·사실        |
| `pace`                | 느리고 관찰적  | 빠르고 사건 중심    |
| `emotionalIntensity`  | 절제됨      | 감정적으로 강함     |
| `narrativeComplexity` | 직선적      | 다층적·퍼즐형      |
| `ambiguity`           | 명확한 해답   | 열린 해석        |
| `toneDarkness`        | 밝고 편안함   | 어둡고 무거움      |
| `spectacle`           | 인물·대화 중심 | 시각적 규모·액션 중심 |
| `conventionality`     | 실험적·비정형  | 익숙한 장르 문법    |

### 발견·문화 관련 보조 변수

| 변수                   | 의미                  |
| -------------------- | ------------------- |
| `mainstreamAffinity` | 대중작과 마이너 작품 선호      |
| `criticalAffinity`   | 평론가형과 관객형 선호        |

| `noveltySeeking`     | 안전한 추천과 의외의 추천 선호   |

## 16유형은 없애야 하는가

없애기보다는 **내부 계산과 외부 놀이를 분리**하는 것이 좋습니다.

```text
내부 추천 엔진
= 8~14개의 연속형 변수

사용자에게 보여주는 결과
= 이해하기 쉬운 MVTI 유형명 1개 + 보조 성향 2개
```

예:

```text
당신의 MVTI
깊게 파고드는 세계관 탐험가

주요 성향
- 복잡한 서사 84
- 어두운 톤 76
- 마이너 작품 탐색 69
```

즉, 16유형은 추천 엔진의 진실이 아니라 **공유 가능한 인터페이스**가 됩니다. 이 구조가 Buddy 바이럴에도 훨씬 유리합니다.

# 9. MVTI Buddy 바이럴 연구

이 주제도 별도 연구가 필요하지만, 반드시 7A 이후에 해야 합니다. Buddy를 찾으려면 “취향이 같다”를 어떤 수학적 기준으로 판단할지 먼저 확정해야 하기 때문입니다.

바이럴 구조의 핵심은 단순 공유가 아닙니다.

```text
내 결과 생성
→ 공유 카드 발급
→ 친구가 링크로 테스트
→ 두 사람 취향 유사도 공개
→ 같이 볼 영화 추천
→ 다른 친구 초대
```

공유 문구 예시는 다음과 같습니다.

> 나와 영화 취향 85% 이상 같은 사람 찾습니다.
> 당신과 내가 같이 봐도 싸우지 않을 영화는?

Buddy 결과에는 다음을 보여줄 수 있습니다.

* 취향 일치율
* 가장 비슷한 성향 3개
* 가장 다른 성향 1개
* 둘 다 좋아할 영화
* 한 명은 좋아하고 한 명은 싫어할 위험 영화
* 함께 보면 토론이 생길 영화

마케팅은 이후 다음 네 영역으로 나눠 연구해야 합니다.

1. 결과 카드와 공유 문구
2. 친구 비교 링크와 초대 루프
3. SNS별 콘텐츠 포맷
4. 초기 사용자 확보와 커뮤니티 시딩
