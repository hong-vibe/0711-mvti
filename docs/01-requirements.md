# MVTI 앱 요구사항 정의서 (`01-requirements.md`)

## 1. 개요 및 사용자 관점 정의

| 질문 | 답변 |
| --- | --- |
| **이 앱은 누구를 위한 것인가요?** | 자신의 성격(MBTI)과 실제 영화 취향(MVTI) 간의 심리적 보완/일치 간극을 분석하고, 매일 맞춤 영화 추천을 받고자 하는 대중 사용자 |
| **사용자는 어떤 정보를 기록하고 싶나요?** | 자신이 시청한 영화에 대한 주관적인 선호 반응(좋아요/싫어요/보통), 선호 강도, 시청 상태(봤음/보고 싶음/지금은 아님), 개인 메모 |
| **목록 화면에서 어떤 정보가 보여야 하나요?** | 평가한 영화의 포스터, 제목, 주 장르, 선호도 상태, 선호 강도, 작성한 메모 내용, 생성/수정일자 |
| **사용자는 어떤 값을 입력하나요?** | 온보딩 단계: 자신의 MBTI 유형 (혹은 '잘 모르겠어요')  <br>반응 등록 단계: 영화 선택 후 선호도, 반응 강도(1~3단계), 관람 상태, 한줄 평/메모 |
| **수정할 수 있는 값은 무엇인가요?** | 이미 등록된 영화 반응의 선호도, 반응 강도, 관람 상태, 한줄 평/메모 내용 (영화 제목/정보 자체는 수정 불가) |
| **삭제 시 사용자가 실수하지 않도록 어떤 확인이 필요한가요?** | 삭제 단추 클릭 시 "정말 이 영화의 반응 기록을 삭제하시겠습니까?"를 명확히 묻는 확인 대화상자(Confirm Modal) 노출 |
| **데이터가 하나도 없을 때 어떤 안내문을 보여줄까요?** | "아직 등록된 영화 반응 기록이 없습니다. 영화를 추가하여 당신의 MVTI 취향을 분석해보세요!" 메시지와 함께 검색/추가 페이지 이동 버튼 제공 |

## 2. 앱 한 줄 소개

> 이 앱은 **영화 관람객**이 자신의 **영화 선호 반응 기록**을 **조회/추가/수정/삭제**하고, 이를 분석하여 **개인화된 영화 취향 유형(MVTI)** 및 원래 성격(MBTI)과의 간극 분석 결과를 도출하도록 돕는 React CRUD 웹 애플리케이션입니다.

## 3. 요구사항 정의표

| 분류 | 요구사항 ID | 기능명 | 상세 설명 | 필수 여부 |
| --- | --- | --- | --- | --- |
| **필수** | F-REQ-01 | 영화 반응 목록 조회 (Read) | 저장된 영화 반응 목록을 카드 형식으로 노출. 포스터, 반응, 강도, 메모 표시. | 필수 |
| **필수** | F-REQ-02 | 영화 반응 추가 (Create) | 영화를 검색/선택한 후 선호도(좋아요/싫어요 등)와 강도, 메모를 적어 반응 기록을 추가. | 필수 |
| **필수** | F-REQ-03 | 영화 반응 수정 (Update) | 목록의 개별 항목 수정 단추 클릭 시 모달창에서 반응 강도, 관람 상태, 메모를 편집 후 저장. | 필수 |
| **필수** | F-REQ-04 | 영화 반응 삭제 (Delete) | 목록에서 삭제 단추 클릭 후 확인창을 거쳐 항목을 영구 삭제. | 필수 |
| **필수** | F-REQ-05 | LocalStorage 영속성 저장 (Persist) | 데이터 생성/수정/삭제 시 LocalStorage와 실시간 동기화하여 브라우저 새로고침 후에도 유지. | 필수 |
| **권장** | R-REQ-01 | 컴포넌트 아키텍처 분리 | 단일 파일 작성을 지양하고 App, Form, List, Card, Dialog 등으로 역할 분리. | 권장 |
| **권장** | R-REQ-02 | 입력 유효성 검증 | 반응 등록/수정 시 필수값(선호도) 누락이나 중복 영화 추가에 대해 가드라인 제공. | 권장 |
| **권장** | R-REQ-03 | 빈 상태 및 오류 처리 | 기록이 없을 때 빈 화면(Empty State) 렌더링, LocalStorage 파싱 실패 시 예외 처리. | 권장 |
| **도전** | D-REQ-01 | MVTI 취향 판정 엔진 | 4대 축(F-R, H-L, E-I, O-C) 계산식을 통해 사용자의 MVTI 4글자 유형 및 분석 신뢰도 계산. | 도전 (반영) |
| **도전** | D-REQ-02 | MBTI↔MVTI 간극 해석 | SDD.txt에 제시된 `GAP_INTERPRETATION_DB`를 통해 사용자의 성격과 영화 소비 성향 대조 해석. | 도전 (반영) |
| **도전** | D-REQ-03 | 날짜 시드형 오늘의 3편 추천 | 날짜 정보와 취향 데이터에 기반해 로컬에서 3가지 슬롯(정확, 확장, 의외) 영화를 계산해 추천. | 도전 (반영) |
| **제외** | N-REQ-01 | 외부 서버 통신 및 회원 가입 | 불필요한 API 노출 및 백엔드 지연 방지를 위해 로컬 정적 데이터 연산으로 완성 (Supabase는 후순위 보류). | 제외 |

## 4. 기능 단위 분해표 (CRUD)

| 기능 단위 | 입력값 | 처리 로직 | 출력 형태 | 확인 방법 |
| --- | --- | --- | --- | --- |
| **목록 조회 (Read)** | LocalStorage 내 반응 배열 | 데이터가 없으면 Empty UI 렌더링, 있으면 생성 최신순으로 정렬하여 카드 리스트 출력 | 반응 카드 목록 (영화 포스터, 정보, 별도 메모 및 반응 아이콘) | 새로고침(F5) 후 목록 상태 유지 확인 |
| **항목 추가 (Create)** | movieId, sentiment, strength, watchStatus, note | 중복 검사 ➔ 유효성 체크 ➔ 기존 데이터 배열의 복사본 끝에 새 Reaction 객체 병합 ➔ LocalStorage 기록 | 목록 하단 또는 최상단에 추가된 새 카드 렌더링 | 입력 값 없이 추가 버튼 클릭 시 경고창 노출, 정상 입력 후 즉시 목록 추가 확인 |
| **항목 수정 (Update)** | reactionId, 변경할 값 객체 (strength, note 등) | 해당 reactionId 객체를 찾아 복사본 업데이트 ➔ React State 변경 ➔ LocalStorage 갱신 | 수정 완료된 카드 내용 실시간 반영 | 카드 내 수정 버튼 ➔ 폼 모달 입력 ➔ 저장 후 변경 확인, 취소 시 기존 상태 유지 검증 |
| **항목 삭제 (Delete)** | reactionId | 확인 대화상자 노출 ➔ 긍정 응답 시 `filter`를 사용해 기존 배열에서 해당 ID 제외 ➔ LocalStorage 갱신 | 목록에서 해당 영화 반응 카드 제거 | 삭제 단추 ➔ 확인창 "확인" ➔ 카드 목록 개수 차감 및 로컬스토리지 삭제 상태 검증 |
| **데이터 유지 (Persist)** | 반응 목록 데이터 배열 | state가 변할 때마다 LocalStorage key `mvti_user_reactions_v1`에 stringify하여 백업 | 새로고침 후 데이터 복구 및 렌더링 | 추가/수정/삭제 후 새로고침하여 직전 변경사항이 완벽하게 불러와지는지 체크 |
